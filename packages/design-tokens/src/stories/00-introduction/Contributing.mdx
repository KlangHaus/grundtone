# Contributing to Haspen UI

Thank you for your interest in contributing to Haspen UI! This guide will help you get started with contributing to the design tokens package and the broader design system.

## Ways to Contribute

There are many ways to contribute to Haspen UI:

### 🐛 Report Bugs
Found a bug? Help us fix it by creating a detailed bug report.

### 💡 Suggest Features
Have an idea for a new token, function, or improvement? We'd love to hear it.

### 📖 Improve Documentation
Documentation can always be better. Help us make it clearer and more comprehensive.

### 🎨 Add Design Tokens
Propose new colors, spacing values, or other tokens that would benefit the system.

### 🔧 Fix Issues
Browse open issues and submit pull requests to fix bugs or implement features.

### ✅ Write Tests
Help us improve code quality by writing tests for untested code.

## Getting Started

### 1. Fork and Clone

Fork the repository on GitHub and clone your fork:

```bash
git clone https://github.com/YOUR_USERNAME/haspen-ui.git
cd haspen-ui
```

### 2. Install Dependencies

Install dependencies using pnpm:

```bash
pnpm install
```

> **Note**: We use pnpm for this project. Please don't use npm or yarn.

### 3. Create a Branch

Create a new branch for your changes:

```bash
git checkout -b feature/your-feature-name
# or
git checkout -b fix/bug-description
```

Branch naming conventions:
- `feature/` - New features
- `fix/` - Bug fixes
- `docs/` - Documentation updates
- `refactor/` - Code refactoring
- `test/` - Test additions or changes

### 4. Make Changes

Make your changes following our coding standards (see below).

### 5. Test Your Changes

Run tests to ensure everything works:

```bash
# Run all tests
pnpm test

# Run tests in watch mode
pnpm test:watch

# Run type checking
pnpm typecheck

# Run linting
pnpm lint

# Build packages
pnpm build:packages
```

### 6. Commit Your Changes

We use [Conventional Commits](https://www.conventionalcommits.org/):

```bash
git add .
git commit -m "feat(design-tokens): add new spacing token"
```

Commit message format:
```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

**Valid types:**
- `feat` - New feature
- `fix` - Bug fix
- `docs` - Documentation changes
- `style` - Code style changes (formatting)
- `refactor` - Code refactoring
- `test` - Test changes
- `chore` - Build/tooling changes

**Valid scopes for design-tokens:**
- `design-tokens` - Design tokens package
- `core` - Core functionality
- `docs` - Documentation
- `ci` - CI/CD

### 7. Create a Changeset

For changes that affect published packages, create a changeset:

```bash
pnpm changeset
```

Follow the prompts to:
1. Select which packages changed
2. Choose the version bump type (major/minor/patch)
3. Write a summary of changes

The changeset will be included in your commit and used to generate changelog entries.

### 8. Push and Create PR

Push your branch and create a pull request:

```bash
git push origin feature/your-feature-name
```

Then create a pull request on GitHub with:
- Clear title and description
- Reference to any related issues
- Screenshots (if applicable)
- Checklist of what was done

## Coding Standards

### TypeScript

- Use strict TypeScript
- Provide type definitions for all public APIs
- Avoid `any` types
- Use descriptive variable names

```typescript
// ✅ Good
interface ColorToken {
  name: string;
  value: string;
  shade?: number;
}

// ❌ Bad
interface Token {
  n: string;
  v: any;
}
```

### SCSS

- Use BEM naming convention
- Leverage design tokens
- Comment complex logic
- Keep specificity low

```scss
// ✅ Good
@use '@ipeeon/design-tokens' as tokens;

.button {
  background-color: tokens.getColor('primary');

  &--large {
    padding: tokens.spacing('lg');
  }
}

// ❌ Bad
.button {
  background-color: #0059b3;
}
```

### File Organization

- One token category per file
- Keep related tokens together
- Use clear, descriptive file names
- Export everything through index files

```
design-tokens/
├── src/
│   ├── core/
│   │   ├── colors.scss
│   │   ├── typography.scss
│   │   └── spacing.scss
│   ├── functions/
│   │   ├── colors.scss
│   │   └── spacing.scss
│   └── index.ts
```

### Documentation

- Document all public APIs
- Provide usage examples
- Include TypeScript types
- Explain the "why" not just the "what"

```typescript
/**
 * Get a color from the palette
 *
 * @param name - Color name (e.g., 'blue', 'primary')
 * @param shade - Color shade from 50-900 (default: 500)
 * @returns Hex color value
 *
 * @example
 * ```scss
 * .button {
 *   background: getColor('blue', 500); // #0059b3
 * }
 * ```
 */
@function getColor($name, $shade: 500) {
  // Implementation
}
```

## Design Token Guidelines

### Adding New Tokens

When proposing new design tokens:

1. **Justify the Need**: Explain why the token is needed
2. **Follow Naming Conventions**: Use consistent, semantic names
3. **Provide Examples**: Show how it would be used
4. **Consider Accessibility**: Ensure WCAG compliance
5. **Test Across Use Cases**: Verify it works in various contexts

### Token Naming

Follow these naming conventions:

```scss
// Colors
$color-primary: #0059b3;
$color-blue-500: #0059b3;

// Spacing
$spacing-md: 1rem;
$spacing-4: 1rem;

// Typography
$font-size-base: 1rem;
$font-weight-bold: 700;

// Semantic names over literal names
$color-error: $color-red-500; // ✅ Good
$color-red: #dc3545; // ❌ Less clear
```

### Color Tokens

- Maintain 10-shade palettes (50-900)
- Ensure WCAG AA contrast ratios
- Provide both palette and semantic colors
- Document accessibility guidelines

### Spacing Tokens

- Use multiples of 4px (0.25rem)
- Provide both numeric and semantic names
- Cover common use cases
- Keep the scale consistent

### Typography Tokens

- Use a modular scale (1.125 or 1.200)
- Provide font families with fallbacks
- Include weight and line-height variants
- Consider readability across sizes

## Testing Requirements

All contributions should include appropriate tests:

### Unit Tests

```typescript
import { describe, it, expect } from 'vitest';
import { COLORS } from '../src';

describe('Color Tokens', () => {
  it('should export primary color', () => {
    expect(COLORS.primary).toBe('#0059b3');
  });

  it('should have 10 shades for each color', () => {
    expect(Object.keys(COLORS.blue)).toHaveLength(10);
  });
});
```

### Visual Regression Tests

For changes affecting visual output, include Storybook stories for visual regression testing with Chromatic.

### Accessibility Tests

Ensure color combinations meet WCAG standards:

```typescript
it('should meet WCAG AA contrast ratio', () => {
  const ratio = getContrastRatio(COLORS.primary, COLORS.white);
  expect(ratio).toBeGreaterThanOrEqual(4.5);
});
```

## Pull Request Process

1. **Update Documentation**: Add or update docs for your changes
2. **Add Tests**: Include tests for new functionality
3. **Run Checks**: Ensure all tests and lints pass
4. **Create Changeset**: Add changeset for versioning
5. **Request Review**: Tag maintainers for review
6. **Address Feedback**: Respond to review comments
7. **Merge**: Maintainers will merge when approved

### PR Checklist

Before submitting, ensure:

- [ ] Code follows style guidelines
- [ ] Tests pass (`pnpm test`)
- [ ] Linting passes (`pnpm lint`)
- [ ] Type checking passes (`pnpm typecheck`)
- [ ] Documentation is updated
- [ ] Changeset is created (if needed)
- [ ] Commit messages follow conventions
- [ ] PR description is clear and complete

## Code Review

All contributions go through code review:

### What We Look For

- **Code Quality**: Clean, maintainable code
- **Testing**: Adequate test coverage
- **Documentation**: Clear documentation
- **Accessibility**: WCAG compliance
- **Performance**: No unnecessary overhead
- **Compatibility**: Works across supported browsers

### Review Timeline

- **Initial Response**: Within 2 business days
- **Full Review**: Within 1 week
- **Follow-ups**: Within 2 business days

## Release Process

Releases are automated using Changesets:

1. Maintainers merge PRs with changesets
2. Changeset bot creates a "Version Packages" PR
3. Maintainers review and merge the version PR
4. GitHub Actions automatically publishes to npm

You don't need to worry about versioning—just create a changeset!
